# The Unofficial Guide — Project 1

A Student Technology Survival Guide: a retrieval-augmented (RAG) question-answering system
over campus IT documentation, built with sentence-transformers, ChromaDB, and the Groq API.

---

## Setup & Usage

**1. Create the environment and install dependencies**

```bash
python3.12 -m venv .venv312
source .venv312/bin/activate
pip install -r requirements.txt
```

**2. Add your Groq API key**

```bash
cp .env.example .env
# then edit .env and set GROQ_API_KEY=...  (free key at https://console.groq.com)
```

**3. Add your source documents**

Drop your collected sources into `documents/` as `.txt`, `.md`, `.html`, or `.pdf` files.

**4. Run the pipeline (in order)**

```bash
python ingest.py      # documents/  -> chunks.json   (load, clean, chunk)
python embed.py       # chunks.json -> chroma_db/     (embed + store vectors)
python retrieve.py "How do I reset my password?"   # optional: test retrieval only
streamlit run app.py  # launch the chat UI in your browser
```

Re-run `ingest.py` then `embed.py` whenever you add or change documents.

## Domain

My system is a **Student Technology Survival Guide** — a question-answering assistant for the
everyday campus-technology problems students hit: resetting passwords, connecting to campus
Wi-Fi, installing Microsoft 365, enrolling in multi-factor authentication, using Canvas, and
spotting phishing emails.

This knowledge is valuable because it is real, recurring, and time-sensitive — a student
locked out of their account the night before a deadline needs an answer in seconds. The
problem is that the answers are **scattered**: some live on the university IT site, some in
Microsoft's support pages, some in a downloadable PDF, and some only in the lived experience of
other students on forums. No single official page covers all of it, and the official pages that
do exist are often buried under navigation, written for IT staff, or out of date. This system
centralizes those sources into one searchable guide that returns a direct, cited answer instead
of making the student hunt across a dozen tabs.

## Chunking Strategy

**Chunk size:** 180 words (fixed, applied to all sources)

**Overlap:** 30 words (~17%)

**Preprocessing before chunking:** HTML sources are stripped of `<script>`, `<style>`,
`<nav>`, `<header>`, `<footer>` elements with BeautifulSoup; PDFs are extracted with
pdfplumber; all text then has whitespace and blank lines normalized (`clean_text` in
[ingest.py](ingest.py)).

**Why these choices fit your documents:** My embedding model, `all-MiniLM-L6-v2`, truncates
any input longer than **256 tokens** (roughly 180–200 words). If I used larger chunks, the tail
of each chunk would be silently dropped before it was ever embedded — so a 500-word chunk would
only be searchable on its first ~180 words. Fixing the chunk size at **180 words** keeps every
chunk fully inside the model's window, so nothing is lost. My documents are mostly short
procedural how-tos (a password reset is ~6 steps), and 180 words is enough to hold one complete
procedure. The **30-word overlap** (~17%) carries a sentence or two across each boundary, so a
step that lands at the end of one chunk isn't orphaned from the step that continues into the
next. I originally considered variable per-source sizing (smaller for FAQs, larger for forum
posts), but since the larger tiers would have exceeded the 256-token limit and been truncated
anyway, a single fixed size was both simpler and more honest about what the model can actually use.

**Final chunk count:** 58 chunks across 10 source files (avg ~163 words/chunk;
min 35, max 180).

## Embedding Model

**Model used:** `all-MiniLM-L6-v2` (sentence-transformers), with cosine similarity in ChromaDB.

**Production tradeoff reflection:** I chose `all-MiniLM-L6-v2` because it is small, fast, runs
locally with no API cost, and is more than accurate enough for short IT how-to text — which made
it ideal for developing and testing this project. If I were deploying this for real Xavier
students and cost were not a constraint, I would weigh moving to a larger model with a longer
context window (e.g. an OpenAI or Cohere embedding API, or a bigger sentence-transformers model
like `all-mpnet-base-v2`). The main thing I'd gain is **accuracy on domain-specific terms** —
"MFA," "eduroam," "OneDrive," "Banner PIN" — and a longer context window that would let me use
larger chunks without truncation, which directly caused a limitation in this build. The costs
I'd weigh against that are **latency** (a hosted API adds a network round-trip per query),
**ongoing per-call cost**, and **data-privacy** concerns about sending student questions to a
third party. For a low-traffic student guide, I'd likely keep a local model but upgrade to a
larger one rather than pay per-query for a hosted embedding service.

## Grounded Generation

**System prompt grounding instruction:** The system prompt in [generate.py](generate.py)
instructs the model to (1) answer *only* using the numbered sources in the context,
(2) cite each fact as `[Source N]`, and (3) say it doesn't have the information and point
to the IT Help Desk rather than invent details when the context doesn't cover the question.
Generation runs at `temperature=0.2` to favor faithful over creative answers.

**How source attribution is surfaced in the response:** Retrieved chunks are passed to the
model as a numbered, source-labeled context block (`[Source 1] (filename) ...`), the model
cites those numbers inline in its answer, and the Streamlit UI lists each source below the
answer in an expander showing the originating file, chunk index, and similarity score.

## Failure Case Analysis

My system has **two** questions that fail, and they fail for *different* reasons at the same
pipeline stage (ingestion) — which together make the limitation clear.

### Failure case 1 — Q5: laptop recommendations (source never collectable)

**Question that failed:** Evaluation Q5 — *"What laptop is recommended for college students?"*

**What the system returned:**
> "I don't have enough information on that. The provided sources do not mention recommended
> laptop specifications for college students, except for a brief mention of 'Suggested
> Computer Specs' in [Source 2], but it does not provide the actual specifications. I suggest
> contacting the IT Help Desk for assistance."

The top retrieved chunks were `microsoft365-install.txt` (similarity 0.31), `xula-student-resources.txt`
(0.30), and `canvas-faq-students.txt` (0.25) — all far below the 0.55–0.70 similarities seen
for questions the corpus *does* cover (e.g. password reset retrieved at 0.57).

**Root cause (tied to a specific pipeline stage):** This is an **ingestion-stage coverage
(recall) gap**, not a generation failure. The laptop-recommendation source was planned as a
Reddit community thread (planning.md rows 11–15), but Reddit blocks automated fetching, so it
was never collected into `documents/`. With no laptop-spec document in the corpus, retrieval
can only return the *nearest* chunks, which are semantically unrelated (hence the low
similarity scores). Notably, the embedding step did surface a tempting near-miss — a chunk
mentioning "Suggested Computer Specs" — but that text names the heading without the actual
specs, so even the best-matching chunk lacked the answer. Generation then behaved exactly as
designed: the grounding instruction caused it to **refuse rather than hallucinate** plausible
laptop advice from the model's training data.

**What you would change to fix it:** Fix it at the source, where the gap actually is —
manually capture a real laptop-recommendation thread (copy a specific `old.reddit.com/.../comments/...`
thread into a `.txt`, or add a non-Reddit "best laptops for students" guide) and re-run
`ingest.py` + `embed.py`. As a secondary safeguard, I could add a **minimum-similarity
threshold** in [retrieve.py](retrieve.py): if the top chunk scores below ~0.35, short-circuit
to the "I don't have enough information" response before even calling the LLM — turning the
low-similarity signal we observed here (0.31) into an explicit, cheaper guardrail.

### Failure case 2 — Q2: campus Wi-Fi (substitute removed for being misleading)

**Question that failed:** Evaluation Q2 — *"How do I connect my laptop to campus Wi-Fi?"*

**What the system returned:**
> "I don't have enough information on that. The provided sources do not contain information about
> connecting to campus Wi-Fi. I recommend contacting the IT Help Desk for assistance."

The top retrieved chunks were `canvas-faq-students.txt` (0.35), `xula-password-guide.pdf` (0.29),
and `microsoft365-install.txt` (0.27) — again well below the 0.55–0.70 of covered questions.

**Root cause (tied to a specific pipeline stage):** This is also an **ingestion-stage gap, but
with a different origin than Q5, and one I caused on purpose.** Early on I *did* have a Wi-Fi
document — a Brown University eduroam guide used as a substitute because no reachable XULA Wi-Fi
page exists (`password.xula.edu` and `www2.xula.edu` don't resolve; the XULA ITC pages contain
only navigation text). With that file indexed, Q2 returned a fluent, well-grounded answer — but
the answer instructed XULA students to choose **"Brown University"** as their organization and
sign in with **`@brown.edu`**. That exposed a subtle failure mode: *retrieval and generation
both "worked" perfectly, yet the answer was wrong for my actual users* because the **source
itself was wrong for the domain.** A high similarity score and a clean citation do not
guarantee a correct answer if the underlying document is mismatched. I judged that more harmful
than admitting ignorance, so I removed the Brown file — which is why Q2 now honestly refuses.

**What you would change to fix it:** The real fix is a domain-correct source: capture XULA's
actual Wi-Fi instructions (likely only available by contacting the ITC Help Desk, or as an
internal doc) and index that. The broader lesson I took from this is a **source-quality check at
ingestion** — substitute/representative sources should be tagged in metadata and, ideally,
surfaced to the user as "general guidance, not XULA-specific," so a confident answer from a
mismatched source can't silently mislead.

## Spec Reflection

**One way the spec helped you during implementation:** Writing the chunking strategy and the
architecture diagram in planning.md *before* coding gave the AI tool a concrete target, so the
generated code matched my intent on the first pass instead of being a generic RAG template. For
example, because planning.md named the exact embedding model (`all-MiniLM-L6-v2`) and committed
to a custom `chunk_text()` function, the ingestion script was built around the model's real
256-token limit and produced the precise 180-word / 30-word chunks I specified — rather than the
arbitrary character-split a vaguer prompt would have produced. The evaluation-question table in
the spec also doubled as my test suite: I ran those exact questions back through the finished
system, which is how I found the Q5 coverage gap.

**One way your implementation diverged from the spec, and why:** My original Documents list
assumed I could pull from 15 specific URLs (university homepages and Reddit threads), but in
practice those sources didn't survive contact with reality: `password.xula.edu` didn't resolve,
the handbook and library pages returned 404s, the homepages contained only navigation/marketing
text, Reddit blocked automated fetching, and Canvas's site was JavaScript-only. I diverged by
switching to **deep-linked official documentation** (the real XULA password PDF, Microsoft's MFA
and phishing support articles) and **representative university IT knowledgebases** (Brown,
UW-Madison) as stand-ins where a XULA-specific page wasn't reachable. This kept the corpus
substantive and answerable, but it introduced a real tradeoff I documented honestly: the
Microsoft 365 answer is correct in its *steps* but contains *Brown-specific* details (login
domain, URL) rather than XULA's — visible in my evaluation report. I drew a line on this: when
the Wi-Fi substitute went further and actively told students to choose "Brown University" and
log in with `@brown.edu`, I judged that too misleading to keep and **removed it**, letting the
Wi-Fi question honestly return "I don't have enough information" instead. I updated the
planning.md Documents table to record the actual sources and substitutions so the spec and
implementation stay consistent.

## AI Usage

**Instance 1 — Generating the ingestion + chunking code**

- *What I gave the AI:* My planning.md Documents section, Chunking Strategy section, and the
  pipeline diagram, and asked it to implement a script that loads my documents, cleans them, and
  produces chunks matching my chunk size and overlap.
- *What it produced:* A `chunk_text()` function plus loaders for `.txt/.md/.html/.pdf` and a
  `clean_text()` step. It first pointed out that my Chunking Strategy fields were actually
  *blank* and that my architecture diagram listed tools I wasn't using (PyPDF2, OpenAI, FAISS)
  instead of my real stack (pdfplumber, sentence-transformers, ChromaDB, Groq).
- *What I changed or overrode:* I rejected the variable per-source chunk sizes from my own draft
  analysis once the AI flagged that `all-MiniLM-L6-v2` truncates past ~256 tokens — I set a single
  fixed **180-word / 30-word** chunk instead, and had it correct the architecture diagram to match
  my actual tools. I also directed it to attach source-file metadata to every chunk, because
  source attribution was a requirement in my plan.

**Instance 2 — Building a real document corpus when my planned URLs failed**

- *What I gave the AI:* My list of 15 planned source URLs and a request to fetch them and save
  them as `.txt` files.
- *What it produced:* It fetched them, but showed me that the results were almost entirely
  navigation menus and listing pages (e.g. the IT homepage returned "Skip to main content … MENU
  CLOSE …"), and that several URLs were dead or blocked. It then searched for and fetched
  deep-linked replacements — the real XULA password PDF, Microsoft MFA/phishing articles, and
  university IT-KB pages — and reported word counts so we kept only substantive content.
- *What I changed or overrode:* I directed it **not** to index the boilerplate fetches and to
  delete them, and chose the "find correct deep links" path over copy-pasting by hand. I also
  decided to **keep the un-answerable laptop question (Q5) in my evaluation** rather than hide it,
  using it as my documented failure case, and had it record every substitute source in
  planning.md so my documentation stayed honest about what is and isn't XULA-specific.
